\section{Experimental Setup}

We evaluate each chunking strategy using the unified framework described in Section~\ref{sec:benchmarking_document_segmentation_strategies}. This section details the specific implementations, datasets, and evaluation procedures used in our experiments.

\subsection{Segmentation Methods}
To ensure a fair comparison, we use a consistent implementation for each segmentation strategy across all datasets.

For \textbf{structure-based heuristics}: the \textbf{Paragraphs} strategy splits documents at newline characters and removes empty segments. The \textbf{Fixed-Size} method divides documents into uniform 256-token segments with no overlap. The \textbf{Sentences} approach uses a regex-based splitter and groups five sentences per chunk by default, following the implementation from LateChunking~\cite{latechunking2024}.

For \textbf{semantically-informed and LLM-guided strategies}, the \textbf{Semantic Sentence} strategy employs llama-index's \texttt{SemanticSplitterNodeParser} with \texttt{jina-embeddings-v2-small-en}, also following the LateChunking work~\cite{latechunking2024}, to compute cosine similarity between adjacent sentences and insert boundaries based on similarity thresholds. For the \textbf{Proposition-Based} strategy, we utilize \texttt{Gemini-2.5-Flash} for processing (implementation details in supplementary materials). The \textbf{LumberChunker} strategy uses a configurable generator system with \texttt{Gemini-2.5-Flash} for fair comparison across LLM-based methods. For all LLM-based methods, we set the generation temperature to 0.0 to ensure deterministic output.

\subsection{Embedding Models Used}

To evaluate the effectiveness of different segmentation strategies and embedding paradigms, we select the following four models:
\begin{itemize}
	\item \textbf{jina-embeddings-v2-small-en(Jina-v2)~\cite{jina2}}: A lightweight model optimized for efficiency while maintaining competitive performance.
	
	\item \textbf{jina-embeddings-v3(Jina-v3)~\cite{jina3}}: An updated version with improved multilingual capabilities and semantic understanding.
	
	\item \textbf{nomic-embed-text-v1(Normic-v1)~\cite{nomic}}: A model designed for high-quality dense retrieval with strong performance on diverse benchmarks.
	
	\item \textbf{multilingual-e5-base(E5-base)~\cite{e5}}: A multilingual model using mean pooling for text representation.
\end{itemize}

While the first three models were evaluated in the original LateChunking work~\cite{latechunking2024}, we additionally include multilingual-e5-base to test the robustness of our approach. Since LateChunking requires models that use average pooling for representation, our selection was constrained to the highest-performing embedding models on the MTEB leaderboard that employ mean pooling for text representation.

All embedding models handle the \textit{pre-embedding} paradigm, where documents are first chunked and then each chunk is encoded separately. For \textit{contextualized chunking}, which requires encoding multiple chunks together, we employ a maximum context utilization strategy: we feed the maximum possible number of chunks that fit within the embedding model's context window. When the document exceeds this limit, we create a new context window for the remaining chunks.

\subsection{Datasets and Evaluation}
Our evaluation is designed to test the generalizability of chunking strategies across different retrieval tasks and domains.

\subsubsection{In-Document Retrieval}
We evaluate in-document retrieval using the \textbf{Guten-\\QA} dataset~\cite{duarte_lumberchunker_2024}, which contains 100 classic literature books. For each book, a set of questions asks for specific information contained within that book. Overall, there are 3,000 question-answer pairs derived from 100 books. This task tests for precise information localization within a single, long document.

The ground truth in GutenQA is annotated at the paragraph level. Since chunks may contain partial paragraphs or span multiple paragraphs, we consider a retrieved chunk as relevant if it overlaps with any ground-truth paragraph. We evaluate retrieval performance by examining whether the top-n retrieved chunks contain the relevant information.

For evaluation, we report DCG@10 (Discounted Cumulative Gain at rank 10), following exisiting work from LumberChunker~\cite{lumber}, calculated as:
$$ \mathrm{DCG@10} = \sum_{i=1}^{10} \frac{2^{\mathrm{rel}_i} - 1}{\log_2(i+1)} $$
where $\mathrm{rel}_i$ is the relevance score of the result at rank $i$.

\subsubsection{In-Corpus Retrieval}
We evaluate in-corpus retrieval on a selection of datasets from the \textbf{BEIR} benchmark~\cite{thakur_beir_2021}: FiQA, ArguAna, SciDocs, TREC-COVID, SciFact, and NFCorpus. We chose these specific datasets based on two criteria. First, they cover a diverse range of retrieval tasks and domains, from finance and scientific articles to biomedical texts. Second, these datasets have a relatively small number of documents, which was a practical consideration to manage the computational cost associated with the LLM-guided chunking methods.

For in-corpus retrieval, ground truth is annotated at the document level. Following established practices~\cite{dai2019deeper}, we use the score of the highest-ranked chunk from each document as the document's relevance score. This max-pooling approach has been shown to be the most effective strategy for aggregating chunk-level scores to document-level rankings.

For evaluation, we report nDCG@10 (normalized Discounted Cumulative Gain at rank 10) for all BEIR tasks. The nDCG@10 metric normalizes the DCG@10 score by the ideal DCG@10 score, producing a value between 0 and 1.
