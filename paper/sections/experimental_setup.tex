\subsection{Datasets}

We evaluate all chunking and embedding methods across both in-document and in-corpus retrieval benchmarks. For in-document retrieval, we use GutenQA~\cite{duarte_lumberchunker_2024}, a long-context question answering dataset designed to test retrieval within book-length documents. For multi-document retrieval, we adopt a diverse subset of BEIR benchmarks: FiQA, ArguAna, SciDocs, TREC-COVID, GutenQA, SciFact, and NFCorpus.
These datasets span heterogeneous document types and retrieval challenges, providing a robust testbed for evaluating the generality of chunking strategies.

\subsection{Chunking Methods}

We implement both traditional and semantically informed segmentation methods.

\subsubsection{Traditional Methods}
\begin{itemize}
    \item Fixed-size: Documents are divided into windows of 512 tokens with no overlap.
    \item Paragraph-based: Documents are segmented using natural paragraph delimiters (newline).
    \item Sentence-based: Each chunk contains five consecutive sentences.
\end{itemize}
    
\subsubsection{Semantic and LLM-Guided Methods}
\begin{itemize}
    \item Semantic Sentence-based: implemented using the jina-embeddings-v2-small-en~\cite{gunther_jina_2024} model via the LlamaIndex~\cite{Llamaindex} framework with default parameters, identifying sentence boundaries based on semantic similarity.
    \item LumberChunker: for GutenQA, chunk boundaries are determined using Gemini-1.5-Flash~\cite{team_gemini_2024}; for BEIR, we use GPT-4.1-Nano~\cite{noauthor_introducing_nodate} following the official LumberChunker methodology.
    \item Proposition-based: for GutenQA, propositions are generated by first applying LumberChunker to identify rough semantic units, followed by proposition extraction using GPT-4.1-Nano. For BEIR datasets, propositions are generated directly using GPT-4.1-Nano without an intermediate step.
\end{itemize}

\subsection{Embedding Models}

We employ four open-source dense embedding models, all of which employ mean pooling to aggregate token-level representations into chunk-level embeddings. Mean pooling is essential for late chunking since contextual embeddings are aggregated over token sequences:

\begin{itemize}
    \item jina-embeddings-v2-small-en~\cite{jina_small_v2}
    \item jina-embeddings-v3~\cite{sturua_jina-embeddings-v3_2024}
    \item nomic-embed-text-v1 ~\cite{nussbaum_nomic_2025}
    \item Multilingual E5 ~\cite{wang_multilingual_2024}
\end{itemize}

For \texttt{multilingual-e5-base}, the maximum context window is limited to 512 tokens, and certain inputs may exceed this capacity. These cases are handled as described in Section~\ref{sec:longtext} (\textit{Long-Text Handling}).

We define two encoder configurations for all models. The \textbf{Regular Encoder} corresponds to pre-embedding chunking, where each segment is processed independently. The \textbf{Late Encoder} corresponds to contextualized chunking: the entire document is encoded first, and chunk representations are subsequently aggregated from token embeddings. When models require special tokens (e.g., [CLS], [SEP], or instruction prefixes), prefix tokens are assigned to the first chunk, suffix tokens to the last, and intermediate chunks correspond strictly to the original text span.

\subsection{Long-Text Handling}
\label{sec:longtext}

Handling long documents is critical since all embedding models impose finite input-length constraints. For pre-embeddings methods,chunk sizes are predefined to remain within model limits, so truncation rarely occurs. When a segment exceeds the limit, excess tokens are conservatively truncated to ensure compatibility with the encoder.

For contextualized chunking methods, no chunking before input sequences may exceed the maximum model length. Following the Late Chunking framework~\cite{gunther_late_2025}, long documents are first split into partially overlapping spans, each fitting within the model’s input window. Each span is then encoded independently to obtain contextual token embeddings, after which segmentation is applied within each span. Prefix tokens are preserved in the first chunk, suffix tokens in the last, ensuring consistent alignment between embeddings and the original text.

\subsection{Evaluation Metrics}
For GutenQA, we report \textbf{DCG@10} following prior work on long-context retrieval in~\cite{lumberchunker}. For BEIR datasets, we adopt \textbf{nDCG@10}, the standard metric for evaluating multi-document retrieval performance~\cite{latechunking}. All evaluations are conducted using identical retrievers and ranking pipelines to isolate the impact of segmentation strategies.
